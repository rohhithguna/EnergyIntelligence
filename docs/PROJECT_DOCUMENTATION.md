# EnergyIntelligence - Full Project Documentation

## 1. Project Overview

EnergyIntelligence is a Python-based analytics pipeline for time-series flow monitoring. It:

- Loads CSV data.
- Cleans and normalizes records.
- Detects pattern classes using clustering.
- Detects spikes and drops using moving-average thresholds.
- Detects anomalies using Isolation Forest.
- Computes numeric correlations.
- Generates human-readable insights.
- Visualizes results with Matplotlib.

The pipeline is orchestrated by `main.py`.

## 5. End-to-End Pipeline Flow

`main.py` executes this flow in sequence:

1. `load_data("sample_data.csv")`
2. `preprocess_data(df)`
3. `run_core_engine(df)`
   - pattern detection (KMeans)
   - spike/drop detection (moving average thresholding)
4. `detect_anomalies(df)` (IsolationForest)
5. `compute_correlation(df)`
6. `generate_insights(spikes, drops, anomalies, correlation_matrix)`
7. Print summaries to console
8. Plot timeline with markers for spikes, drops, and anomalies

DataFrame mutation strategy:

- A copy is created in pattern detection.
- Several downstream steps mutate DataFrame columns in-place (`moving_avg`, `anomaly`).

## 6. Module and Function Reference

### 6.1 `data/loader.py`

#### `load_data(file_path)`
Purpose:
- Reads CSV data into a pandas DataFrame.

Behavior:
- Calls `pd.read_csv(file_path)`.
- Returns DataFrame as-is.

Input requirements:
- Valid file path to CSV.

Output:
- DataFrame with at least columns expected by downstream modules (`time`, `data_rate` in current sample).

---

### 6.2 `data/preprocessor.py`

#### `preprocess_data(df)`
Purpose:
- Cleans and standardizes raw loaded data before analytics.

Operations in order:

1. Remove fully empty rows: `dropna(how='all')`
2. Parse time column to datetime: `pd.to_datetime(errors='coerce')`
3. Remove rows where time conversion failed: `dropna(subset=['time'])`
4. Fill missing `data_rate` forward (`ffill`) then backward (`bfill`)
5. Remove duplicates: `drop_duplicates()`
6. Keep non-negative rates only: `df[df['data_rate'] >= 0]`
7. Sort by time ascending

Output:
- Cleaned DataFrame suitable for numeric processing and time plotting.

Important compatibility note:
- The code currently uses `.ffill()` and `.bfill()`. This is compatible with current pandas behavior.

---

### 6.3 `core/pattern_spike_engine.py`

This module encapsulates pattern labeling plus spike/drop event detection.

#### `_get_target_col(df)`
Purpose:
- Selects supported numeric signal column.

Rules:
- If `data_rate` exists, use it.
- Else if `usage` exists, use it.
- Else raise `ValueError`.

Why it exists:
- Supports two possible schema names for signal column.

#### `detect_patterns(df)`
Purpose:
- Assigns categorical flow labels using KMeans clustering.

Detailed behavior:

1. Copy input DataFrame (`out = df.copy(deep=True)`).
2. Determine target column via `_get_target_col`.
3. Coerce target column to numeric; invalid becomes 0.0.
4. Handle empty DataFrame by creating empty `cluster` column.
5. Determine cluster count:
   - max of 1
   - min of 3, number of rows, and number of unique values
6. Fit KMeans (`random_state=42`, `n_init=10`).
7. Get cluster centers and sort center values.
8. Map sorted centers to semantic labels in increasing order:
   - low flow
   - medium flow
   - high flow
9. Save mapped label in `cluster` column.

Output:
- DataFrame copy with `cluster` label column added.

#### `detect_spikes_and_drops(df)`
Purpose:
- Detects abrupt high and low events relative to rolling baseline.

Logic:

1. Ensure signal numeric, fill invalid with 0.0.
2. Compute moving average with `window=3`, `min_periods=1`.
3. Spike rule: value > 1.5 * moving_avg
4. Drop rule: value < 0.5 * moving_avg
5. Exclude cases where moving_avg <= 0
6. Return index positions of spikes and drops.

Outputs:
- `spikes`: list of DataFrame index values
- `drops`: list of DataFrame index values

#### `run_core_engine(df)`
Purpose:
- Wrapper that runs both pattern detection and spike/drop detection.

Returns:
- `out` DataFrame with pattern + moving average columns
- `spikes` list
- `drops` list

---

### 6.4 `core/anomaly_engine.py`

#### `detect_anomalies(df, contamination=0.05)`
Purpose:
- Flags rare events using Isolation Forest.

Model:
- `IsolationForest(contamination=0.05, random_state=42)`

Behavior:

1. Fit model on `df[['data_rate']]`.
2. Save model output into `df['anomaly']`.
   - `1` means normal
   - `-1` means anomaly
3. Slice anomalies as `df[df['anomaly'] == -1]`.
4. Return updated DataFrame and anomaly-only DataFrame.

Input assumptions:
- `data_rate` column exists and is numeric/coercible.

---

### 6.5 `analysis/correlation.py`

#### `compute_correlation(df)`
Purpose:
- Computes correlation matrix for numeric columns.

Behavior:
- `df.corr(numeric_only=True)`

Output:
- Correlation matrix DataFrame for all numeric columns currently present.

Notes:
- Datetime columns like `time` are excluded because `numeric_only=True`.

---

### 6.6 `analysis/insight_generator.py`

#### `generate_insights(spikes, drops, anomalies, correlation_matrix)`
Purpose:
- Converts technical outputs into text insights.

Current rule set:

1. If spikes exist -> add overload-style insight.
2. If drops exist -> add interruption/failure-style insight.
3. If anomalies exist -> add count-based anomaly insight.
4. Correlation insight check:
   - Attempts to read correlation between `time` and `data_rate` if both are in correlation matrix columns.
   - If present and > 0.5 -> positive trend insight.
   - If present and < -0.5 -> negative trend insight.
   - Else -> weak/no strong correlation insight.
5. If no rules fire -> stable system message.

Important practical note:
- Since correlation matrix is numeric-only, `time` usually will not be present, so this branch often does not run.

Output:
- `insights`: list of human-readable strings.

## 9. Data Model and Schema

Expected core columns in DataFrame:

- `time`:
  - source type: string timestamp from CSV
  - preprocessor converts to pandas datetime
- `data_rate`:
  - source type: numeric with occasional missing values
  - cleaned by forward/backward fill

Derived columns added by pipeline:

- `cluster` (string): low/medium/high flow label
- `moving_avg` (float): rolling average window=3
- `anomaly` (int): IsolationForest prediction (`1` or `-1`)

Optional compatibility column:

- `usage` (alternative signal column name supported only in pattern/spike module)

## 12. Design Choices and Implications

### 12.1 Hybrid rule + ML approach

- KMeans provides unsupervised pattern buckets.
- Threshold logic gives deterministic spike/drop events.
- IsolationForest adds anomaly sensitivity beyond simple thresholds.

Result:
- Pipeline is interpretable and relatively lightweight.

### 12.2 Mutation and side effects

Some functions mutate incoming DataFrames (`detect_spikes_and_drops`, `detect_anomalies`).

Implication:
- Useful for fast prototyping.
- Requires awareness when reusing the same DataFrame object elsewhere.

### 12.3 Correlation strategy

- Numeric-only correlation avoids type errors.
- Time trend insight currently depends on `time` being numeric in matrix, which is generally not true with current implementation.

Implication:
- Time-trend text insight is often skipped.

## 13. Known Gaps and Improvement Opportunities

1. Dependency declaration missing:
- No `requirements.txt` or `pyproject.toml` included.

2. Error handling is minimal:
- Missing column or malformed CSV cases are not deeply validated in all modules.

3. Inconsistent target-column support:
- Pattern module supports `usage` fallback.
- Anomaly and plotting logic assume `data_rate` only.

4. Insight correlation check mismatch:
- Looks for `time` in correlation matrix even though matrix is numeric-only.

5. Plot interactivity assumptions:
- `plt.show()` may block or warn on headless systems.

6. No automated tests currently present.
